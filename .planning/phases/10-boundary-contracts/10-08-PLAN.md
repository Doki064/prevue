<objective>
Close the three P0 gaps from 10-UAT.md's live sandbox verification, all boundary-contract breakages in the reusable workflow's public surface that unit tests could not catch because they test the ladder logic in isolation, not the actual `workflow_call` wiring a real consumer hits:

1. Copilot's OTEL-based real-token capture is provably non-functional on the pinned `@github/copilot@1.0.61` CLI (confirmed via live sandbox run + official docs check: zero mention of OTEL/telemetry in `copilot --help` or GitHub Copilot CLI docs; `~/.copilot` and the OTEL directory are never created). Flip `usage_capture` to `"none"` so Copilot reports an honest `~est` estimate instead of silently claiming real capture that never fires.
2. The declared CONFIG_PRECEDENCE "workflow input" tier for the review model is architecturally unreachable — `workflow_call.inputs` has no `model` field and the Run review step never forwards `PREVUE_MODEL`/`COPILOT_MODEL`. Add the missing input + env passthrough so `resolve_review_model()`'s already-correct env-wins logic actually receives a value from a real consumer.
3. `jobs.review.outputs` is declared at the job level but GitHub Actions requires an additional top-level `on.workflow_call.outputs:` block to re-map those same keys for a caller's `needs.<job>.outputs.*` — without it, every job output comes back empty across the `workflow_call` boundary (live-confirmed via a downstream probe job).

Purpose: These are all instances of the same failure class — internal logic is correct and unit-tested, but the public workflow-YAML surface never reaches it. Fixing the YAML wiring (not the Python logic, which was already right) closes the loop.
Output: Updated `.github/workflows/prevue-review.yml` (+ `prevue-command-run.yml` parity), `src/prevue/engines/spec.py` (Copilot spec flip), regression tests for all three gaps, and docs brought in line with what's actually reachable.

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Consumer `with: model` input → `PREVUE_MODEL` env → engine CLI `--model`/`COPILOT_MODEL` argv | A public-facing config knob now reachable from any caller workflow; must not allow the model value to smuggle shell metacharacters or flags |
| Reusable workflow → caller's `needs.<job>.outputs.*` | Values are plain scalars (counts, cost, conclusion string) already validated/serialized by `emit_machine_output`; no new untrusted-input surface introduced by re-mapping them |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-08-24 | Tampering | `inputs.model` → `PREVUE_MODEL` env → CLI `--model`/`-m` argv flag | accept | Value is passed as a single argv element (never through a shell), matching the existing `--model`/`-m` argv construction already used for `engine.model`/`engine.models.review` from yml — no new injection surface versus the already-accepted yml-sourced model string. Consumer already controls their own workflow `with:` block; this is a self-inflicted-only vector (a consumer can already set `PREVUE_MODEL` today via repo Variables + a custom caller workflow — this task only makes the *documented* input path work) |
| T-10-08-25 | Information Disclosure | `on.workflow_call.outputs` re-mapping `jobs.review.outputs.*` | accept | Re-exposes exactly the same 7 scalar values (severity counts, cost, conclusion, schema_version) already written to `$GITHUB_OUTPUT` by the existing, security-reviewed `emit_machine_output` (Phase 10 Plan 05) — no new data is exposed, only a missing propagation path is completed |
