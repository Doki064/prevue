<objective>
Close two UAT gaps from `10-UAT.md` (Phase 10 Boundary Contracts live verification):

**Gap A (cursor-cli real token usage, UAT test 4, severity major):** `spec.py` invokes `cursor-agent` with `--output-format text`, so even though prevue's usage-capture dispatcher exists, no JSON envelope is ever requested. Live research against the official Cursor CLI docs (`cursor.com/docs/cli/reference/output-format`) confirms the `--output-format json` envelope schema is `{type, subtype, is_error, duration_ms, duration_api_ms, result, session_id, request_id}` — **no token/usage/cost fields exist**. Separately, `junhoyeo/tokscale` (the user's cited "Cursor is definitely possible" reference) does NOT read `cursor-agent` stdout at all — it reads Cursor's authenticated web billing API via a manually-extracted session cookie, a completely different mechanism outside an engine adapter's scope. Conclusion: the confirmed bug (wrong output format requested) gets fixed; the token-fields claim is now a verified fact (no fields exist), not a stale unverified research note — `usage_capture` honestly stays an estimate, but via the now-correct JSON code path (reusing the exact same envelope-unwrap logic already proven for Claude Code) instead of the disconnected `"none"` strategy.

**Gap B (antigravity-cli headless auth, UAT test 1, severity major):** Per the official Antigravity CLI docs (`antigravity.google/docs/cli/install`), there is no API-key / non-interactive auth mode. `spec.py`'s `secret_env=ANTIGRAVITY_API_KEY` (+ `GEMINI_API_KEY` alias in `cli_adapter._build_env`) was a design-time bet on a LOW-confidence third-party citation (10-RESEARCH.md Assumption A1) that was never closed by the `checkpoint:human-verify` gated in `10-06-PLAN.md` Task 3. No credential or code fix makes `agy` authenticate headlessly if the mechanism doesn't exist. The honest fix is the mechanism Phase 10 already built for exactly this situation: flip `functional=False` (D-03) so `require_functional_adapter("antigravity-cli")` fails closed with a clear error instead of silently no-op'ing or crashing mid-review, and stop documenting it as a selectable engine.

Purpose: Both fixes replace unverified assumptions with research-confirmed reality, and make the system fail closed (informative error) rather than fail open (silent wrong behavior) where the underlying capability does not exist.
Output: `cursor-cli` requests and correctly parses its real JSON envelope (still estimated, now provably so); `antigravity-cli` is registered but excluded from `require_functional_adapter`, with tests and docs updated to match.
</objective>
