## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sticky dismiss block → suppress-list | The persisted JSON is bot-authored on the base-ref conversation surface; parsing must fail safe |
| maintainer reason text → sticky render | Untrusted free text rendered into the human-visible audit |
| PR head → suppress-list | MUST NOT be a source — the list is read only from the trusted sticky |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-22 | Tampering | malicious / corrupt dismiss block over-suppressing or crashing (Pitfall 4) | mitigate | parse_dismiss_block fails safe to [] on any malformed JSON or invalid entry; DismissEntry extra="forbid"; bounded by review.max_dismissals; round-trip + tolerant-parse tests. |
| T-08-23 | Tampering | a PR-head file used to inject suppressions (D-15 guard 4) | mitigate | The block lives in the bot-authored sticky only; render/parse touch the sticky body exclusively; never read from PR-head content (architectural — no PR-head read path exists in this plan). |
| T-08-24 | Spoofing | a forged sticky body faking dismissals | mitigate | Sticky provenance is enforced by the shipped _is_trusted_sticky_actor; only the bot can author the trusted sticky read by run_review. |
| T-08-25 | Tampering | reason free-text breaking sticky markdown / HTML | mitigate | Human-visible reason escaped via _escape_table_cell / _neutralize_html; JSON block is fenced and verbatim. |
