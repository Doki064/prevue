<objective>
Close gap #3 (minor, framework-context leak) and gap #4 (minor, noop install overhead),
both touching `.github/workflows/prevue-review.yml`; gap #3 also touches the cursor
engine adapter.

Gap #3: the Run review step uses `working-directory: .prevue` and `cursor_cli.py`'s
`subprocess.run` (cursor_cli.py:46-52) passes no `cwd=`, so cursor-agent inherits the
.prevue framework checkout as its workspace. On PR #23 the first-run sticky described
the Prevue Python/GitHub-Actions codebase instead of the consumer repo. Fix: run
cursor-agent with `cwd=` the consumer root (the workflow already exports
`PREVUE_CONSUMER_ROOT = ${{ github.workspace }}/consumer`, prevue-review.yml:104).

Gap #4: a same-SHA noop re-run still runs checkout + uv sync + engine CLI install
(~40s) before the Python noop is detected inside `prevue review`. Add a workflow
pre-flight step that compares the sticky marker head SHA to the PR head SHA and skips
the engine CLI install (and ideally the heavy setup) when they are equal.

Purpose: keep engine review prose scoped to the consumer repo (correctness +
isolation) and avoid paying the full install cost on a no-op re-run (latency).

Output: cwd-isolated cursor invocation + a noop-aware workflow pre-flight gate, with
tests for both. Permissions stay minimal.

No file overlap with plan 08-08 (review.py / comments.py / review-flow & comments
tests), so this plan runs in wave 1 alongside 08-08.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| .prevue framework checkout → cursor-agent workspace | UNTRUSTED isolation boundary: framework code must NOT be the engine's review target (info-leak + wrong-context) |
| consumer PR head context → engine cwd | The consumer repo is the intended (still-untrusted) review target; cwd scopes the engine to it |
| sticky comment marker → pre-flight noop decision | Untrusted PR comment content parsed for the marker SHA; a spoofed marker only affects whether install is skipped (fail-safe: a wrong skip still runs `prevue review`, which re-derives state) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-10-01 | Information Disclosure | cursor-agent cwd at .prevue | mitigate | Set `cwd=PREVUE_CONSUMER_ROOT` so the engine reads/describes the consumer repo, not the Prevue framework checkout (closes the PR #23 leak) |
| T-08-10-02 | Elevation of Privilege | pre-flight gh api call | mitigate | Pre-flight uses GITHUB_TOKEN read scope only (sticky comment read); permissions block stays contents: read / pull-requests: write / checks: write — no new scope |
| T-08-10-03 | Tampering | spoofed sticky marker forcing install skip | accept | Worst case a forged `head=<PR head>` marker skips the engine CLI install; `prevue review` still runs and re-derives scope from the live diff/comments — no false-green, no review suppression beyond the noop path it would take anyway |
| T-08-10-04 | Tampering | pull_request_target / secrets exposure | mitigate | Trigger and fork guard unchanged (`head.repo.full_name == github.repository`); no `pull_request_target`, no `secrets: inherit` (asserted by test) |
| T-08-10-SC | Tampering | npm/pip/cargo + curl installer | mitigate | No new package added; the existing cursor official-curl-installer supply-chain guard (08-07) remains; do not weaken it when adding the `if:` gate |
</threat_model>
