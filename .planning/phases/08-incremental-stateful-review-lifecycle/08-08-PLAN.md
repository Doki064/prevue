<objective>
Close gap #1 (the only blocker, major/LIFE-02): the sticky Findings table diverges
from the live inline comments when the engine re-reports the same `(path, line)`
with a DIFFERENT normalized title (a new fingerprint). Today `_open_set_findings`
(`review.py:147-162`) drops the carried prior whenever ANY current finding shares
its `(path, line, side)`, so `render_findings_table(gate.placed)` shows the NEW
engine title — while `post_inline_review` (`comments.py:685-691`) skips the inline
edit by location (D-06 quiet-by-default), leaving the OLD title on the live thread.
Result: sticky index title ≠ live open comment title, violating the LIFE-02 spirit
("never drop still-valid comments") for the human-facing overview.

Purpose: make the sticky Findings overview a faithful index of the live open inline
comments on incremental carry-forward, and make the keep-vs-edit decision consistent
between the two code paths (fingerprint match per D-06, not bare location).

Output: corrected `_open_set_findings` reconciliation + a fingerprint-aligned
skip/keep decision in `post_inline_review`, with the existing test that encodes the
buggy behavior FLIPPED and new tests asserting title preservation.

This is a TDD plan (RED → GREEN → REFACTOR). The RED phase includes flipping
`test_open_set_dedupes_carried_prior_at_same_line_as_current` (currently asserts the
current/engine title wins) to assert the carried prior title is preserved.
</objective>

    Decision contract (the chosen fix — keep the carried prior in the open-set when
    the inline was NOT updated, AND align skip-edit with D-06 fingerprint match):

    For a carried PriorFinding P and the current findings C at the same (path, line, side):
    - If some current finding c in C has the SAME fingerprint as P
      (fingerprint(P.path, P.title) == fingerprint(c.path, c.title)): true duplicate —
      drop the carried prior (current wins; identical content). This is the existing
      `prior.fingerprint in current_fps` branch and must remain.
    - If NO current finding at that location shares P's fingerprint (rephrase-at-same-line,
      a DIFFERENT normalized title): the inline thread keeps P's old title (D-06 quiet),
      so the OPEN-SET must ALSO keep P (the carried prior) so the sticky Findings row
      shows the SAME title the live inline shows. The new engine finding at that location
      is NOT separately listed (one open inline thread per (path,line,side) — the live
      thread is P). Result: exactly one open-set entry for that location, carrying P's title.
    - At a location with current findings but NO carried prior: behave as today
      (current finding(s) deduped by location, highest severity).
    - At a location with a carried prior and NO current finding: keep the carried prior
      (already correct — `test_open_set_keeps_carried_prior_at_different_line`).

    post_inline_review skip/keep alignment (D-06): the keep-as-is vs edit decision at
    `comments.py:685-691` must be made on FINGERPRINT match (prior title fingerprint ==
    current finding title fingerprint) gated by severity escalation, NOT on bare
    (path,line,side) presence. When the prior and current fingerprints DIFFER at the
    same location (rephrase), keep the existing inline as-is (no edit, no new comment) —
    consistent with the open-set keeping the prior. Do not create a duplicate inline at
    the same (path,line,side).
