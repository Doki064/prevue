<objective>
Build the deterministic, untrusted-input-safe foundation of the `/prevue`
command surface (D-16): a pure command parser, the `issue_comment` event-context
loader, and the authoritative write-access authorization gate. No dispatch / side
effects yet (those land in 08-15) — these are the isolated, heavily-tested units
the highest-risk surface in the project rests on.

Purpose: The locked decision authorizes on `author_association`, but research
(Fact 5) shows that is NOT a reliable write-access proxy — the authoritative
gate must be `get_collaborator_permission ∈ {write, admin}` (strengthening
D-16's write-access intent). The `issue_comment` payload has no PR head SHA
(Fact 3 / §L1), so a NEW `load_comment_context` sibling loader is mandatory —
`load_pr_context` is left untouched. Treating the comment body as untrusted data
parsed in Python (never shell) is the core injection mitigation (Fact 4).
Output: `parse_command`, `authorize_commenter`, `load_comment_context`, and the
`issue_comment` fixture.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| comment body → parse_command | Attacker-writable text; must be data parsed in Python, never shell/eval |
| commenter login → authorization | Write access is a server-side fact via the permission API, not author_association |
| issue_comment payload → PR resolution | Payload has no head SHA; PR fetched via get_pull (no PR-head checkout) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-30 | EoP | unauthorized user triggers a privileged command | mitigate | authorize_commenter gates on get_collaborator_permission ∈ {write,admin}; author_association is advisory only (Fact 5). HIGH per security_block_on — verified live in 08-16 §L7 checkpoint. |
| T-08-31 | Tampering/EoP | shell / argument injection via comment body (Fact 4) | mitigate | Body parsed in pure Python (parse_command); strict grammar; id tokens validated against [0-9a-f]{16} / [A-Za-z0-9_=-]{8,}; no eval/shell=True; injection battery tests. |
| T-08-32 | DoS / crash | reusing load_pr_context on issue_comment → KeyError (Pitfall 2 / §L1) | mitigate | New sibling load_comment_context reads issue/comment keys + get_pull; load_pr_context untouched; KeyError test. |
| T-08-33 | EoP | fork PR resolving to base-context execution (SECR-01) | mitigate | load_comment_context exposes head_repo_full vs base_repo_full for the fork guard consumed by dispatch (08-15); no PR-head checkout anywhere on this path. |
| T-08-34 | Spoofing | author_association spoof granting access | mitigate | Association never authorizes on its own; the permission API is authoritative (Fact 5 / CONTRIBUTOR edge case test). |
| T-08-SC | Tampering | npm/pip/cargo installs | accept | No new packages (RESEARCH-LIFE0305 §Standard Stack: zero new installs). |
</threat_model>
