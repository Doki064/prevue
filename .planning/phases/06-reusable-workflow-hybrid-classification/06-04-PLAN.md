<objective>
Deliver the skip-pipeline vertical slice (NOIS-01) and close the phase with a
live consumer-repo verification (WKFL-01/02 success criterion #1).

Python-side skip evaluation (`should_skip`) detects bot authors
(`pr.user.type == "Bot"`, re-includable via `skip.review_bots`), the default
`skip-review` label, and configurable `skip_labels` / `skip_title_patterns`
(D-14/D-15). On a skip, `run_review` posts a NEUTRAL `prevue/review` check plus a
sticky note stating the reason — reusing the empty-PR neutral-skip surface
(D-16) — so a required-check branch protection rule is never left pending
(Pitfall 5). Draft skip already lives at the workflow `if:` (06-02); this plan
adds the Python skips that need config and a posted check.

After this plan: bot/labelled/title-filtered PRs are declined-with-a-neutral-check
by default, and a separate sandbox consumer repo proves end-to-end adoption.

Purpose: Closes NOIS-01 ("skip drafts, bots, title/label-filtered PRs by default,
configurable") and provides the only verification that proves WKFL-01 success
criterion #1 — a separate repo adopting at a pinned ref — which no unit test can
cover.

Output: new `src/prevue/skip.py`; extended `github/comments.py` + `github/checks.py`
(reason + neutral conclusion); `run_review` skip hook; green `tests/test_skip.py`
+ extended `tests/test_review_flow.py`; a live sandbox-consumer PR verification.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PR author metadata → skip decision | pr.user.type/login, labels, title drive the skip; an attacker controls their own PR's title/labels |
| skip outcome → required-check branch protection | A skip must not silently bypass a required check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-14 | Tampering | required-check bypass via skip | mitigate | Skip posts a `neutral` `prevue/review` check (not absent, not `success`) — non-blocking but present; required checks never hang and are never falsely passed (Pitfall 5, D-16); test asserts conclusion == "neutral" |
| T-06-15 | Spoofing | bot-author skip evasion / over-skip | mitigate | Generic `pr.user.type == "Bot"` detection (covers any App), `review_bots` is the explicit re-include exception (D-14); not a hardcoded login list |
| T-06-16 | Elevation of Privilege | skip rules read from PR head | mitigate | SkipConfig comes from `.github/prevue.yml` on the consumer base ref (06-01/06-03 config path), never PR head — an attacker cannot rewrite skip rules in their own PR (D-04/D-07) |
| T-06-17 | Repudiation | silent skip | mitigate | Every skip posts a sticky note stating the reason (D-16) — the decision is auditable in the PR |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | No new package installs (RESEARCH audit: zero new deps) |
</threat_model>
