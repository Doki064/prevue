## Phase Goal

**As a** developer who opens a PR that is all noise (e.g. lockfile-only) or whose repo customizes the rules, **I want to** have noise filtered honestly (skipping the engine entirely when nothing is reviewable) and my repo's ignore/label/routing customizations respected additively, **so that** zero tokens are spent on unreviewable PRs and the framework adapts to my repo without losing the built-in protections.

## Acceptance Criteria (this slice)

- [ ] A PR where ALL files are filtered out → neutral skip: no engine call, sticky note "no reviewable files (N filtered)" (D-10).
- [ ] Consumer ignore globs ADD to (do not replace) the built-in noise filters (D-07).
- [ ] Consumer label rules merge additively/override-by-label over built-ins (D-05); consumer routing overrides built-in (D-06 precedence).
- [ ] Absent consumer config → built-in rules unchanged (optional config, RESEARCH Open Question 2).
- [ ] Dropped-file count surfaced in the audit trail (D-09).

Acceptance test asserts malformed consumer YAML raises ValidationError. |
| T-02-09 | Tampering (access control) | malicious consumer config read from untrusted PR head | mitigate | Consumer `prevue.yml` is read from the TRUSTED BASE REF only, never from a checkout of PR code (SECR-01/WKFL-03 posture; ASVS V4). | T-02-10 | Tampering (override abuse) | consumer override suppressing a security label or ignoring security-sensitive files | mitigate (partial) | Override-by-label is additive over built-ins (D-05) — built-in security globs remain unless the consumer explicitly overrides that label; the `general` fallback (D-03, Plan 02) still guarantees no file is silently un-reviewed. Consumer ignore globs are additive (D-07) and the dropped count is disclosed (D-09/D-10), so suppression is visible in the audit trail rather than silent. Deeper per-path override governance is a v2 concern (CUST-01). |
