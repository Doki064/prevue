exactly the security skills my PR's changed files actually trigger (loaded from the
trusted framework, not from the PR), **so that** reviews stay token-efficient and a
malicious PR cannot poison the reviewer's own instructions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PR head content → skill loader | The PR under review must NEVER supply skills; loader reads only the packaged framework dir |
| skill frontmatter → frontmatter parser | YAML parsed on load (SafeLoader); a malformed built-in is a framework bug |
| run_review → engine | Assembled instructions cross into the engine prompt; only trusted framework skill bodies are included |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02 | Tampering / Elevation | `loader.load_skills()` load path | mitigate | SKIL-04 by construction (D-10): resolve root via `importlib.resources.files("prevue.skills")`, never `__file__`/PR head. `test_loads_from_packaged_framework_dir` asserts the invariant. |
| T-03-03 | DoS / Tampering | malformed skill frontmatter | mitigate | D-12 fail-closed: `Skill.model_validate` raises (missing/empty `applies-to` via `min_length=1`); never silently skipped. Verified by `test_missing_applies_to_raises`. |
| T-03-04 | Tampering / RCE | YAML deserialization in frontmatter | mitigate | python-frontmatter uses `yaml.SafeLoader` by default; never switch to full `yaml.load`. |
| T-03-05 | Information Disclosure | `committed-secrets.md` skill guidance | mitigate | Skill body instructs the engine to reference secrets by file+line, NOT echo the full value back into the PR comment (alert-not-redact, no republish). |
| T-03-SC | Tampering | python-frontmatter supply chain | mitigate | Gated in Plan 01 via blocking human-verify; `==1.3.*` pinned; `uv.lock` committed. |
</threat_model>
