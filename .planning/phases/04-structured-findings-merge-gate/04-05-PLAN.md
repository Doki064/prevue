<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GITHUB_TOKEN → Checks API | New write scope (checks: write) exercised by the workflow |
| GateResult → branch protection | The check conclusion controls whether merges are blocked |
| consumer config → run startup | Malformed review config crosses before engine spend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-16 | Elevation of Privilege | workflow permissions | mitigate | `checks: write` added explicitly; exact-equality test pins the full block so any extra scope fails CI (ASVS V4; WKFL-04 groundwork) |
| T-04-17 | Spoofing | check on wrong commit | mitigate | head_sha always from DiffBundle.head_sha (pr.head.sha), never GITHUB_SHA merge commit (Pitfall 3); docstring warning + no GITHUB_SHA reference in checks.py code |
| T-04-18 | Repudiation/Tampering | false merge approval | mitigate | Degrade concludes neutral never success; fork creates NO check (absence holds required checks pending — D-09); only opt-in min_severity_to_fail can conclude failure (D-05/D-06); neutral confirmed non-blocking per official docs (Pitfall 4) |
| T-04-19 | Information Disclosure | PyGithub exceptions in new write paths | mitigate | Check/review write failures propagate through cli.py which prints str(exc) — no tokens appear in PyGithub exception messages for these calls; any subprocess-adjacent text already passes _sanitize_stderr (Phase 1 discipline); avoid logging request payloads |
| T-04-20 | Denial of Service | duplicate check runs on re-run | accept | Same-name check runs on the same SHA are safe — latest wins in the UI (RESEARCH Pattern 4, cited); concurrency group cancels superseded runs |
</threat_model>
